# ADR-0006: ホスティングに Vercel を採用

## ステータス

採用

## コンテキスト

MVP のリリースにあたり、フロントエンド（Next.js）とバックエンド（Hono）のホスティングサービスを選定する必要がある。

要件:

- 無料枠があること（個人プロジェクト・MVP 段階）
- Next.js App Router が動作すること
- GitHub との CI/CD 統合が容易であること
- 将来的な有料プランへの移行パスがあること

## 決定

フロントエンド・バックエンドともに Vercel を採用する。

- フロントエンド（Next.js）: Vercel Hobby プラン
- バックエンド（Hono）: Vercel Hobby プラン（サーバーレス関数）

## 理由

### フロントエンド

Next.js の開発元であり、App Router を含む最新機能への完全対応が保証されている。
他の選択肢（Netlify・Cloudflare Pages）は App Router のサポートが不完全またはプラグイン依存であり、安定性に劣る。

### バックエンド

現状の API 処理（認証・CRUD）はすべて軽量であり、Vercel サーバーレス関数の 10 秒制限に収まる。
フロントエンドと同一プラットフォームに統一することで、デプロイ・環境変数・ログ管理が一元化できる。

将来の AI 分析機能については、処理が長時間化した時点でバックエンドを別サービス（Railway 等）に切り出すか、非同期処理として設計し直す方針とする。

## 検討した選択肢

### 選択肢1: Cloudflare Workers（バックエンド）

- メリット:
  - Hono は Cloudflare Workers 向けに最適化されており相性が最良
  - 無料枠が非常に寛大
  - エッジ実行で低レイテンシ
- デメリット:
  - Prisma が Edge Runtime に非対応（本プロジェクトの ORM として採用済み）
  - bcrypt 等の Node.js ネイティブ API が使用できない
  - 既存スタックの変更コストが大きい

### 選択肢2: Render Free（バックエンド）

- メリット:
  - 常時起動（コンテナ型）
  - Node.js アプリをそのままデプロイ可能
- デメリット:
  - 15 分間アクセスがないとスリープし、初回リクエストに 30〜60 秒かかる
  - MVP の仮説検証フェーズで UX を著しく損なう

### 選択肢3: Railway（バックエンド）

- メリット:
  - 常時起動
  - Node.js アプリをそのままデプロイ可能
  - $5/月のクレジットで小規模なら実質無料
- デメリット:
  - 完全無料ではない（MVP 段階の要件を満たさない）
  - フロントエンドと管理が分散する

### 選択肢4: Fly.io（バックエンド）

- メリット:
  - 常時起動（コンテナ型）
  - 無料枠あり
- デメリット:
  - CLI ベースの設定が複雑で運用コストが高い
  - フロントエンドと管理が分散する

## 影響

### ポジティブ

- フロントエンド・バックエンドのデプロイ・ログ・環境変数を Vercel ダッシュボードで一元管理できる
- GitHub push から自動デプロイまでの設定が最小限で済む
- Next.js の最新機能が安定して利用できる

### ネガティブ

- サーバーレス関数の実行時間が 10 秒に制限される（Hobby プラン）
- 将来の AI 分析機能（長時間処理）追加時にアーキテクチャの見直しが必要になる
- サーバーレスのコールドスタートによる初回レイテンシが発生する場合がある

## 将来の移行方針

AI 分析など長時間処理が必要になった場合は以下を検討する:

- バックエンドを Railway / Render 有料プランに移行
- AI 処理のみを専用のサーバーレスサービス（AWS Lambda 等）に切り出す
- Vercel Pro プラン（実行時間 60 秒）への移行
