# ADR-0002: バックエンドフレームワークに Hono を採用

## ステータス
採用

## コンテキスト
バックエンド API を構築するためのフレームワークを選定する必要がある。
要件:
- TypeScript サポート
- 軽量で高速
- 学習価値が高い（ポートフォリオとしてアピールできる）
- JWT 認証を自前実装しやすい

## 決定
Hono を採用する。

## 理由

### パフォーマンス
- Express の約 4 倍高速（ベンチマーク比較）
- 軽量なランタイムフットプリント

### TypeScript ファースト
- 型定義が最初から組み込まれている
- 型推論が優れており、開発体験が良い

### Web Standards 準拠
- Fetch API、Request/Response などの標準 API を使用
- 学んだ知識が他の環境でも活きる

### マルチランタイム対応
- Node.js、Cloudflare Workers、Deno、Bun で動作
- 将来的な Edge デプロイも視野に入る

### ポートフォリオ価値
- モダンなフレームワークを採用していることをアピールできる
- 「なぜ Express ではなく Hono なのか」を説明できる

## 検討した選択肢

### 選択肢1: Express
- メリット:
  - 最も普及しており情報が多い
  - ミドルウェアエコシステムが充実
- デメリット:
  - 型サポートが弱い（@types/express に依存）
  - 設計が古い（コールバックベース）
  - パフォーマンスが相対的に低い

### 選択肢2: NestJS
- メリット:
  - エンタープライズ向けの堅牢なアーキテクチャ
  - DI（依存性注入）が組み込み
- デメリット:
  - 学習コストが高い
  - 小規模プロジェクトにはオーバーエンジニアリング
  - デコレータベースで独特の書き方

### 選択肢3: Fastify
- メリット:
  - Express より高速
  - プラグインシステムが優秀
- デメリット:
  - Hono ほど軽量ではない
  - Web Standards 準拠ではない

## 影響

### ポジティブ
- 高速な API レスポンス
- TypeScript との優れた統合
- 将来的な Edge デプロイへの道が開ける
- モダンな技術選定としてアピール可能

### ネガティブ
- Express ほど情報が多くない（ただし公式ドキュメントは充実）
- 一部のミドルウェアは自作が必要
