# ADR-0001: モノレポ構成に Turborepo を採用

## ステータス

採用

## コンテキスト

Reflect Forward は フロントエンド（Next.js）とバックエンド（Hono）を持つ Web アプリケーションである。
これらのコードベースをどのように管理するかを決定する必要がある。

選択肢:

1. 単一リポジトリ（モノリス）
2. 複数リポジトリ（マイクロサービス的）
3. モノレポ（1リポジトリで複数パッケージ管理）

## 決定

モノレポ構成を採用し、ツールとして Turborepo を使用する。

## 理由

### モノレポを選んだ理由

- フロントとバックで共通の型定義・バリデーションを共有できる
- 1つの PR でフロント・バック両方の変更をレビューできる
- 個人開発では複数リポジトリの管理は煩雑

### Turborepo を選んだ理由

- Vercel が開発しており、Next.js との親和性が高い
- 設定がシンプル（turbo.json のみ）
- ビルドキャッシュによる高速化
- 学習コストが低い

## 検討した選択肢

### 選択肢1: 複数リポジトリ

- メリット: 独立したデプロイ、チーム分離が容易
- デメリット: 共通コードの共有が困難、管理が煩雑

### 選択肢2: Nx

- メリット: 機能豊富、大規模プロジェクト向け
- デメリット: 設定が複雑、学習コストが高い

### 選択肢3: Lerna

- メリット: 歴史が長く情報が多い
- デメリット: メンテナンスモード、Turborepo の方がモダン

## 影響

### ポジティブ

- `packages/shared` で型定義を共有でき、フロント・バック間の型の不整合を防げる
- CI/CD パイプラインが1つで済む
- コードレビューが効率的

### ネガティブ

- リポジトリサイズが大きくなる可能性
- 部分的なデプロイには工夫が必要
