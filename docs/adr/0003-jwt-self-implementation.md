# ADR-0003: 認証を JWT で自前実装する

## ステータス
採用

## コンテキスト
ユーザー認証機能を実装する必要がある。
選択肢:
1. Supabase Auth を使用（認証を外部サービスに委譲）
2. NextAuth.js を使用
3. JWT を自前で実装

本プロジェクトはポートフォリオとしての価値を最大化することも目的の一つである。

## 決定
JWT (JSON Web Token) を使用した認証を自前で実装する。

## 理由

### 学習価値
- 認証の仕組み（トークン発行、検証、リフレッシュ）を深く理解できる
- パスワードハッシュ化（bcrypt）の実装経験
- セキュリティの基礎知識が身につく

### ポートフォリオ価値
- 面接で「認証を自前実装しました」と言える
- 「JWT の仕組みを説明してください」に答えられる
- セキュリティ意識があることをアピール

### 柔軟性
- 外部サービスへの依存がない
- カスタマイズが自由
- 将来的な要件変更に対応しやすい

## 検討した選択肢

### 選択肢1: Supabase Auth
- メリット:
  - 実装が簡単（数行のコードで完了）
  - Email/Google/GitHub など複数プロバイダ対応
  - セキュリティが担保されている
- デメリット:
  - 認証の仕組みを学べない
  - 外部サービスへの依存
  - ポートフォリオとしての差別化が難しい

### 選択肢2: NextAuth.js
- メリット:
  - Next.js との統合が簡単
  - 多くの認証プロバイダに対応
- デメリット:
  - フロントエンドに認証ロジックが寄る
  - Hono バックエンドとの統合が複雑
  - 「自前実装」ではない

## 影響

### ポジティブ
- 認証フローの完全な理解
- 面接でのアピールポイント
- 外部サービス障害の影響を受けない

### ネガティブ
- 実装に時間がかかる
- セキュリティの責任を自分で負う
- 実装ミスによる脆弱性のリスク

### リスク軽減策
- bcrypt でパスワードハッシュ化（salt rounds: 10 以上）
- JWT の有効期限を短く設定（15分）
- リフレッシュトークンの実装
- HTTPS 必須
- HttpOnly Cookie でのトークン保存

## 実装概要

```typescript
// アクセストークン: 15分
// リフレッシュトークン: 7日
// パスワード: bcrypt (salt rounds: 12)

// トークン構造
{
  sub: "user-id",
  email: "user@example.com",
  iat: 1234567890,
  exp: 1234567890
}
```
